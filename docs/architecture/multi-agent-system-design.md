# 멀티 에이전트 시스템 설계

> **검토 에이전트 1 + 분석 에이전트 3 — 합의 기반 의사결정 시스템**

**최종 수정**: 2026-04-06
**상태**: 설계 완료
**상위 문서**: `PROJECT_CONSTITUTION.md` 제4조

---

## 1. 개요

### 1.1 목적

아이디어 검증, 아키텍처 결정, 보안 변경 시 **단일 에이전트의 편향**을 제거하고 **다관점 합의**를 통해 최선의 결정을 도출한다.

### 1.2 핵심 원칙

1. **독립 분석**: 각 에이전트는 독립적으로 분석 후 결론 도출 (서로 참조 금지)
2. **교차 비교**: 에이전트 간 출력을 비교하여 일치/불일치/누락 식별
3. **합의 도출**: 검토 에이전트가 최종 판단을 종합
4. **투명성**: 모든 에이전트의 의견과 근거가 사용자에게 공개

### 1.3 설계 근거

Anthropic의 연구에 따르면 에이전트는 **자신의 출력을 일관되게 과대평가**한다. 독립적인 평가자를 회의적으로 튜닝하는 것이 생성자를 자기비판적으로 만드는 것보다 훨씬 효과적이다. 이것이 생성과 평가를 분리하는 근본적 이유이다.

### 2.2 에이전트 역할 상세

#### Agent A — 구현 분석가 (Implementation Analyst)

| 항목 | 설명 |
|------|------|
| **관점** | "이것이 실제로 동작하는가?" |
| **초점** | 기술적 구현 가능성, 코드 수준 실현 방법, 의존성 분석, 성능 |
| **출력** | 구현 계획, 기술 스택 호환성, 예상 난이도, 성능 병목 |
| **강점** | 구체적 코드 레벨 분석 |

#### Agent B — 품질/안전성 검증가 (Quality & Safety Auditor)

| 항목 | 설명 |
|------|------|
| **관점** | "이것이 안전하고 견고한가?" |
| **초점** | 보안 취약점, 엣지 케이스, 테스트 커버리지, SDD 정합성 |
| **출력** | 위험 목록, SDD 편차, 보안 체크리스트, 테스트 요구사항 |
| **강점** | 방어적 관점, SDD 명세 대비 편차 탐지 |

#### Agent C — 대안 탐색가 (Alternative Explorer)

| 항목 | 설명 |
|------|------|
| **관점** | "더 나은 방법이 있는가?" |
| **초점** | 대안 기술/접근법, 트레이드오프 비교, 기존 결정에 대한 도전 |
| **출력** | 대안 목록, 비교 매트릭스, 트레이드오프 분석 |
| **강점** | 시야 확장, 건전한 도전 |

#### Reviewer — 검토 에이전트 (Consensus Builder)

| 항목 | 설명 |
|------|------|
| **역할** | 관제탑 — 결과 종합, 최종 판단 |
| **입력** | Agent A/B/C 출력 전체 |
| **처리** | 교차 비교 → 일치/불일치/누락 분류 → 최선 방향 도출 |
| **출력** | 합의 보고서 |
| **권한** | 최종 의사결정권 — 에이전트 간 의견이 갈릴 때 판정 |

## 3. 실행 프로토콜

### 3.1 5-Phase 워크플로우

```
Phase 1: 분배 (Distribution)
─────────────────────────────
메인 컨텍스트가 사용자 요청을 분석하고
각 에이전트에게 관점별 분석 지시를 전달

Phase 2: 독립 분석 (Independent Analysis)  ← 병렬 실행
─────────────────────────────────────────
Agent A, B, C가 동시에 독립 분석 수행
서로의 출력을 참조하지 않음 (편향 방지)

Phase 3: 교차 비교 (Cross-Comparison)
─────────────────────────────────────
검토 에이전트가 3개 출력을 수집하여:
 ① 일치 (Consensus) — 3개 모두 동의
 ② 부분 일치 (Partial) — 2개 동의, 1개 이견
 ③ 불일치 (Divergence) — 3개 모두 다른 의견
 ④ 누락 (Gap) — 특정 에이전트만 언급한 사항

Phase 4: 합의 도출 (Consensus Resolution)
─────────────────────────────────────────
 - 일치: 그대로 채택
 - 부분 일치: 소수 의견의 근거 평가 후 결정
 - 불일치: 각 근거를 비교하여 최선 선택 + 이유 명시
 - 누락: 해당 사항의 중요도 평가 후 포함/제외 결정

Phase 5: 보고 (Report)
──────────────────────
사용자에게 최종 합의 보고서 전달
```

### 3.2 Claude Code 실행 방법

```
1. 메인 컨텍스트가 요청을 분석하고 3개 Agent 도구를 병렬 호출
   → Agent(subagent_type="general-purpose") x 3

2. 각 Agent에게 역할(관점)과 분석 대상을 명시적으로 지시
   → Agent A: "구현 분석가 관점에서 분석하라"
   → Agent B: "품질/안전성 관점에서 분석하라"
   → Agent C: "대안 탐색 관점에서 분석하라"

3. 3개 출력을 수집한 후:
   → 별도 Agent(검토 에이전트)를 호출하여 교차 비교 수행
   → 또는 메인 컨텍스트가 직접 교차 비교 수행

4. 합의 보고서를 사용자에게 제출
```

## 5. 적용 시나리오

### 5.1 아이디어 검증

```
사용자: "새 기능 아이디어: 에이전트가 자동으로 PR을 생성하는 기능"

→ Agent A: 구현 방법, GitHub API 연동, 기술적 복잡도
→ Agent B: 보안 위험(토큰 관리), 잘못된 PR 생성 방지, 테스트 전략
→ Agent C: 기존 도구(Copilot, Cursor)와 비교, 차별화 포인트

→ Reviewer: 교차 비교 → 합의 보고서
```

### 5.2 아키텍처 결정

```
사용자: "이벤트 기반 vs 폴링 기반 에이전트 통신?"

→ Agent A: 각 방식의 구현 복잡도, 인프라 요구사항
→ Agent B: 장애 시나리오, 메시지 유실 방지, 모니터링
→ Agent C: 하이브리드 접근, 기존 사례 분석

→ Reviewer: 교차 비교 → 합의 보고서
```

### 5.3 보안 변경

```
사용자: "API 키 관리 방식을 변경하려고 합니다"

→ Agent A: 새 방식의 구현 가능성, 마이그레이션 경로
→ Agent B: 보안 취약점 분석, OWASP 체크, 기존 시스템 영향
→ Agent C: 다른 비밀 관리 서비스(Vault, AWS Secrets Manager) 비교

→ Reviewer: 교차 비교 → 합의 보고서
```

## 7. 품질 보증

### 7.1 메트릭

| 메트릭 | 측정 방법 | 목표 (설계 단계) | 목표 (구현 세부 검증) |
|--------|----------|----------------|-----------------|
| 합의율 | 3개 모두 동의한 항목 비율 | > 60% | > 20% |
| 유효 이견 비율 | 소수 의견이 채택된 비율 | > 10% | > 30% |
| 누락 탐지율 | 1개 에이전트만 발견한 CRITICAL 이슈 비율 | 추적 | **핵심 지표** |

### 7.2 피드백 루프

```
합의 보고서 → 사용자 확인 → 수정 요청 시:
  ① 어떤 에이전트의 판단이 틀렸는지 기록
  ② 해당 관점의 프롬프트 개선
  ③ 다음 호출에 반영
```

### 7.3 검증 성격별 합의율 패턴 (2026-04-24 관찰)

3+1 프로토콜의 합의율은 **검증 대상의 성격** 에 따라 크게 다르게 나타난다. 관찰된 패턴:

| 검증 대상 | 예시 | 합의율 | 1인 지적 CRITICAL 발생률 |
|---------|------|-------|----------------------|
| **설계 결정** (아키텍처/스택/ADR) | consensus-005 (LLM-judge 안전), consensus-006 (salt rotation), consensus-007 §5 (PR #2 실행 계획) | 40~70% | ~1건/세션 |
| **구현 세부 검증** (함수 배치/에러 surface/persist 필드) | consensus-007 사후 검증 4건 (2026-04-24) | **25%** (1/4) | **1건/4건 = 25%** |
| **PR 범위 vs ADR 명세 충돌** (단계 분할 권고) | consensus-PR-3 (helmet+CSP 분할 3a/3b/3c, 2026-04-28) | **87%** (2+ 일치 가중) / 40% (3 일치 순수) | **0건** (모든 CRITICAL 2+ 일치) |

**해석**:
- 설계 단계는 추상도가 높아 3 에이전트가 유사한 프레임으로 수렴. 합의율 자연 상승.
- 구현 세부는 각 관점(구현/안전/대안)이 **서로 다른 파일/라인을 들여다봄** → 완전 일치보다는 보완 관계. 1인 지적이 결정적이 되는 경우가 많다.
- **PR 범위 vs ADR 명세 충돌** 은 새 패턴으로, ADR 의 정합성을 흔들지 않으면서 PR 을 안전하게 진행하기 위해 Reviewer 가 **ADR 동반 갱신을 권고**하여 "명세 위반" 우려를 해소. consensus-PR-3 사례에서 Agent B 의 "정적 unsafe-inline = ADR 위반 CRITICAL" 우려가 ADR 동시 갱신으로 흡수되어 **위협 분리 (보안 위협 vs 명세 정합성)** 가 명확해짐.

**운영 원칙**:
1. **구현 세부 검증에서 합의율 25% 는 정상 범위** — 설계 합의율과 같은 기준(>60%)으로 판단 금지.
2. **1인 지적 CRITICAL 격상 룰** (Session 4 교훈) 이 구현 세부 검증에서는 더 빈번하게 발동. Reviewer 는 "다수결 경계" 보다 "사실 검증" 에 초점.
3. 누락 탐지율을 구현 세부 검증의 **주 지표** 로 사용. 낮으면 에이전트 프롬프트에서 관점 분리가 부족하다는 신호.

**실증 사례** (`docs/review/consensus-007-session-6-grading-wiring.md` 의 사후 검증):
- Agent B 단독 CRITICAL 격상 1건 (`persistHeldAnswer` 7항 누락 → WORM 감사 체인 갭) — 코드 라인 축자 확인 후 hotfix PR #15 로 승격.
- Agent B HIGH 2건 + Agent C 재배치 권고 1건 — 모두 hotfix PR #15 에 통합.
- Agent A 는 `LlmJudgeTimeoutError` throw surface 의 상위 문서화만 지적 — held persist 필드 데이터 품질은 검증 범위에서 누락. 이는 "구현 관점" 이 "WORM 컬럼 완결성" 을 놓칠 수 있다는 프롬프트 개선 포인트.

### 7.4 Reviewer 의 직접 사실 검증 책임

구현 세부 검증에서 1인 지적 CRITICAL 이 나오면 Reviewer 는 다음을 **직접 수행**:
1. 해당 에이전트가 인용한 파일/라인을 Read 툴로 열어 축자 확인
2. 주장된 위반이 **실제 동작/데이터 경로** 에 영향을 주는지 검증
3. 위반이 확인되면 hotfix 우선순위 (P0/P1/P2) 할당

이를 통해 Agent A/C 가 놓친 문제를 Agent B 가 지적했을 때 **사실 확인을 Reviewer 가 보증**하여 다수결 실수를 방지한다.
