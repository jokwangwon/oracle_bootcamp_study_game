## 주요 의사결정

### 1. promptfoo CLI 대신 standalone 스크립트

promptfoo 0.121.3의 custom provider 형식(file: + config: 조합)이 런타임에서
인식되지 않았고, TypeScript 로더(tsx), 한국어 경로 등 외부 도구 호환 문제가 중첩.

**결정**: promptfoo를 orchestrator로 쓰지 않고, 기존 eval 인프라(assertion 7개 +
schema.v1 + report-generator + promptfoo-adapter의 aggregation)를 직접 호출하는
`run-eval-standalone.ts`를 작성.

- LlmClient → Ollama 직접 호출 (ADR-009 정합)
- 결과는 동일한 `EvalRoundResultV1` schema
- promptfoo 의존 제거로 실행 안정성 확보
- `run-all-models.sh`로 배치화 + `nohup` 지원

### 2. Claude 베이스라인 후순위

Anthropic API 크레딧 미충전 상태. 절대 기준치(C1~C8)만으로도 모델 선정 가능하므로
OSS 모델 평가를 먼저 진행. Claude 베이스라인은 크레딧 충전 후 추가.

### 3. promptfoo.config.yaml 수정

- `baseUrl: http://ollama:11434` → `http://127.0.0.1:11434` (호스트 실행용)
- `model: qwen3-coder:30b` → `qwen3-coder-next:latest` (Ollama 실제 모델명 일치)
- prompts 형식: `id: + file:` 객체 → 단순 문자열 `'file://...'` (promptfoo 호환)

### 4~6. 이전 에이전트 의사결정 (단계 7)

인가는 ENV whitelist / EvalRunnerService 인터페이스 격리 / roundId 형식 /
EvalModule 분리 / 동기 호출 / 에러→HTTP 매핑 — 이전 세션 기록 유지.
