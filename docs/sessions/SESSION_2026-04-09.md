### 1. 운영 일관성 (output-schemas.ts 추출)

운영 `AiQuestionGenerator`가 inline 정의하던 `blankTypingOutputSchema` / `termMatchOutputSchema`를 `apps/api/src/modules/ai/eval/output-schemas.ts`로 추출. AQG가 import 형태로 변경. 평가 MT2 assertion(`zod-schema.ts`)이 동일 schema를 import → **운영과 평가가 같은 schema를 본다**. 한쪽만 갱신되어 평가가 운영을 거짓 예측하는 위험 차단.

### 2. promptfoo 패키지는 단계 5에서 미설치

본 단계의 모든 코드는 promptfoo 패키지에 직접 의존하지 않고 vitest로 단위 검증 가능하게 설계. 이유:

- promptfoo 버전 변경 위험을 본 코드 경계 안에 격리 (provider/assertion contract 타입을 자체 정의).

`npm install --save-dev --workspace=@oracle-game/api promptfoo`는 단계 8 R0 직전에 사용자가 1회 실행. README + CONTEXT.md에 명시.

### 3. promptfoo prompt 형식 — JSON-encoded multi-message

운영 AQG는 `SystemMessage` + `HumanMessage` 두 개를 LlmClient에 보내는데, promptfoo의 prompt는 단일 문자열이라 system을 잃을 위험. 대응:

- `prompts/build-eval-prompt.ts`가 `[{role:"system"}, {role:"user"}]` JSON-encoded 문자열을 반환
- `langfuse-wrapped` provider의 `parsePromptToMessages`가 자동 파싱해 BaseMessage[]로 복원
- 일반 문자열 prompt도 fallback (단일 HumanMessage) 지원

이로써 운영과 평가가 **동일한 메시지 구조**로 LLM에 전달된다.

### 4. 평가 전용 "중심 토큰 강제" 한 줄

운영 prompt는 모델이 화이트리스트 안에서 자유롭게 키워드를 골라 문제를 만든다. 평가에서는 라운드 간 비교 가능성이 필요해 `seedFocusKeyword`를 중심 토큰으로 강제하는 한 줄을 user 메시지 끝에 추가:

```
*** 평가 지시 ***
이번 문제의 정답에는 반드시 `SELECT`가 포함되어야 합니다.
```

term-match는 "정답은 반드시 ... 이어야" 분기. SDD §4.1 Recall 트랙 정의와 정합.

### 5. MT8 sanitization 보수성 (false positive 허용)

`UNION SELECT` 패턴을 SQL injection으로 분류 — 학습용 정상 SQL도 같은 룰로 fail 가능. 의도된 보수성 (SDD §3.1 C8 99% 합격선 안에서 일부 학습 SQL 손실 허용). 테스트로 명시.

# 세션 심야부 — 단계 6 결과 JSON schema + report-generator 완료

## 작업 요약

단계 6: `reports/schema.v1.ts` (감사용 고정 Zod schema, N-05) + `reports/aggregate.ts` (pure stat helpers) + `reports/report-generator.ts` (단일 라운드 + 다중 라운드 비교 markdown) + `reports/promptfoo-adapter.ts` (promptfoo-agnostic RawCallRecord[] 정규화 + raw JSON 파서). TDD 4 모듈 65 신규 cases.

## 주요 의사결정

### 1. schema.v1 immutable 정책

`EVAL_RESULT_SCHEMA_VERSION = 1 as const`. **한번 commit된 v1은 절대 수정 안 함** — 과거 라운드 결과의 호환성 보장. v2가 필요하면 schema.v2.ts를 새로 만들고 promptfoo-adapter / report-generator가 schemaVersion 필드로 dispatch한다. 본 세션 시점에는 v1 단일 schema만 존재.

### 2. 외부 stat 라이브러리 의존 회피

bootstrap CI / Cohen's d / Bonferroni / percentile은 모두 표준 알고리즘. d3-array, jstat 같은 외부 라이브러리를 가져오는 대신 직접 구현 (≈ 200 LOC). 이유:
- 트랜지티브 의존성 비대화 회피
- 단일 컨트리뷰터 환경에서 dependency 감사 부담 최소화
- 결정론(seeded mulberry32 PRNG)을 직접 통제 — 동일 seed = 동일 결과 보장

### 3. promptfoo-agnostic 중간 모양 (RawCallRecord)

promptfoo CLI 출력 JSON shape는 promptfoo 버전에 따라 변할 수 있어, 본 어댑터는:
1. **promptfoo JSON → RawCallRecord[]**: `parsePromptfooRawJson` (promptfoo 0.x 표준 shape 가정, 단계 8 R0 첫 실행 시 실측 검증 필요)
2. **RawCallRecord[] → EvalRoundResultV1**: `aggregateCallRecords` + `buildRoundFromRecords` (promptfoo와 무관, 핵심 비즈니스 로직)

이렇게 분리하면 promptfoo 버전 업그레이드는 (1)만 patch하면 되고, 다른 평가 도구로 교체해도 (1)만 새로 작성하면 된다. 핵심 집계 로직은 promptfoo 결합에서 자유롭다.

### 4. error record는 메트릭 집계에서 제외

Provider error(connection refused 등)가 있는 record는 `errorCount`에 1, 메트릭 pass/total에는 0. 이유: 모델의 품질이 아닌 인프라 문제이므로 pass rate 분모를 왜곡하면 안 된다. SDD §10.4 엣지 케이스 정책과 정합 (timeout/segfault는 별도 카운트).

### 5. 합격선 평가표는 PASS_CRITERIA 상수에 정의

C1~C8 임계값을 `report-generator.ts` 내부 `PASS_CRITERIA` 상수에 코드로 정의. SDD §1.3 변경 시 본 상수도 갱신 (단계 8 Phase 0 baseline 측정 후 ±5%p 재보정 시점). 본 세션 시점은 SDD v2 초기 합격선 그대로.
